    2.0 Work Flow

    2.1 Intro branch workflow

    The Git branching workflow is a strategy for managing and organizing code changes in a collaborative development environment. It allows multiple developers to work on different features or fixes simultaneously without interfering with each other. Here's a explanation of our Git branching workflow.

    The project starts with the branch called master. This branch represents the stable, production-ready version of the code.

    2.1.1 Create a branch

    When a new feature or bug fix needs to be implemented, you create a new branch off the master branch. This branch is named descriptively, such as A2-#YOUR_ISSUE_NUMBER-THE-ISSUE-NAME.

    2.1.2 Add you branch in the issue

    So that we know there is already a branch for the issue and its being worked on

    2.1.3 Work on the Feature Branch

    You make changes, add new code, or fix bugs on the feature branch.

    Commits are made to the feature branch, by keeping in mind the Commit Structure: A2 #YOUR_ISSUE_NUMBER Explain what you did

    2.1.4 Push Feature Branch

    Once you have finished the work, you push it to the feature branch.

    2.1.5 Create a Pull Request (PR)

    After you completed all the work and you think its ready to be added to the master, create a Pull Request, this step allows other team members to review the code changes before adding it to the master.

    2.1.6 Code Review

    Team members review the code in the pull request, provide feedback, and discuss potential improvements. The pull request may go through multiple iterations of review and modification.

    2.1.7 Profit???

    Now just repeat this for each new feature or bug fix, multiple feature branches can coexist, allowing parallel development of different features.

    By using feature branches and pull requests, this workflow promotes collaboration, code review, and a more organized development process. It helps prevent conflicts, facilitates easier bug tracking, and allows for better release management.
